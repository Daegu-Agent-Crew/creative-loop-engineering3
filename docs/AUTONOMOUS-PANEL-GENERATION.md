# 자율 패널 생성 운영 및 인계 가이드

이 문서는 다음 세션이나 다른 에이전트가 Phase 4 패널 생성을 안전하게
이어가기 위한 실행 계약이다. 정책의 기계 판독 원본은
`config/panel-generation-policy.json`이다. 이 문서와 정책이 충돌하면 정책
파일을 우선하고, 충돌 자체를 고성능 모델에 승격한다.

## 목표

- 정상 패널은 저성능 작업 모델이 스스로 선택하고 계속 처리한다.
- 실제 이미지만 `gpt-image-2`가 생성한다.
- 고성능 모델과 사람은 반복 실패와 모호한 판단에만 개입한다.
- 한 패널의 실패가 에피소드 전체 생성을 멈추지 않게 한다.
- 생성 시간과 실패 원인을 기록해 처리량을 측정할 수 있게 한다.

## 세션 시작 체크리스트

1. `AGENTS.md`, 이 문서, `config/panel-generation-policy.json`을 읽는다.
2. 대상 에피소드의 `panels.json`, `generation-jobs.json`, `MANIFEST.md`를 읽는다.
3. `panels/assets/*.png` 실파일과 `generation_status`를 대조한다.
4. 파일이 있는데 상태가 pending이면 덮어쓰지 말고 상태 불일치로 기록한다.
5. 현재 실행 중인 다른 워커가 없는지 확인한다.
6. `node scripts/run-panel-jobs.js --episode EPxxx --dry-run --max-jobs 3`로 다음
   실행 배치를 뽑는다.
7. 준비된 패널만 `ready`로 전환하고 자율 루프를 시작한다.

## 역할 분리

| 역할 | 모델 등급 | 책임 |
|---|---|---|
| 작업 워커 | 저성능 텍스트 모델 | 다음 패널 선택, 복잡도, 참조 이미지, 재시도 판단 |
| 이미지 생성 | `gpt-image-2` | 완성 패널 한 장 생성 |
| 1차 QA | 저성능 비전 모델 | 캐릭터, 스토리보드, 구도, 스타일 검사 |
| 예외 판단 | 고성능 추론 모델 | 반복 실패, 입력 충돌, 모호한 QA 해결 |
| 최종 예외 | 사람 | 최대 시도 초과 또는 정책·안전 판단 |

`low_performance`는 특정 모델명을 고정하지 않는다. 실행 환경에서 사용 가능한
저비용·저지연 모델을 선택하되 JSON 선택과 간단한 시각 판정이 가능한 모델이어야
한다. 이미지 생성 모델은 `gpt-image-2`로 고정한다.

## 스케줄링 규칙

- 배치 의미는 “페이지 이미지를 한 번에 생성”이 아니다. 같은 페이지의 패널들을
  작업 그룹으로 묶고 각 패널은 한 요청당 한 장씩 생성한다.
- 일반 패널은 최대 3개까지 동시에 실행한다.
- 복잡 패널은 최대 1개만 실행한다.
- 전체 실행 중 작업 수는 3개를 넘지 않는다.
- 복잡도는 페이지가 아니라 패널별로 계산한다.
- PNG가 이미 존재하면 자동으로 건너뛰고 상태 대조 대상으로 보낸다.

다음 중 하나면 복잡 패널이다.

- 등장인물 2명 이상
- 참조 이미지 3개 이상
- 몽타주, 풀페이지, 군중, 전투 또는 복잡한 기계 장면
- 이전 생성 실패 이력 존재

## 작업 선택 계약

저성능 모델은 자유 형식 명령을 만들지 않고 다음 형태의 JSON만 반환한다.

```json
{
  "action": "generate",
  "panel_id": "p10-2",
  "complexity": "standard",
  "reference_assets": [
    "episodes/EP001/characters/assets/shiqiang-v1.png"
  ],
  "reason": "필수 입력과 참조 파일이 존재하며 실행 슬롯이 비어 있음"
}
```

허용되는 `action`은 `generate`, `skip_existing`, `retry`, `hold`, `escalate`다.
경로 생성, 파일 이동, 상태 저장과 동시성 제한은 모델이 아니라 결정론적 실행
코드가 담당한다.

## 상태 전이

정상 경로:

```text
pending → ready → running → generated → qa_checking → approved
```

실패 경로:

```text
running → failed → retry_ready → running
                         └──────→ escalated
```

실행기는 패널마다 시도 횟수, 모델 등급, 시작·완료 시각, 생성 시간, QA 점수와
마지막 오류를 기록한다. 성공 시 `panels.json`, `MANIFEST.md`, `state.json`을
같은 작업 단위로 갱신한다. 세 파일 중 하나라도 저장에 실패하면 완료로 확정하지
않고 복구 가능한 오류로 남긴다.

## QA와 재시도

- 42점 이상: 자동 승인
- 35점 미만: 자동 재생성
- 35~41점: 보류 큐로 이동하고 다른 작업은 계속
- 1차 실패: 같은 프롬프트로 재시도
- 2차 실패: 프롬프트를 단순화하고 고성능 모델에 원인 판단 요청
- 3차 실패: 사람 검토 큐로 이동

프롬프트 단순화는 장면의 핵심 인물, 행동, 카메라, 스타일 블록과 출력 규칙을
보존하면서 부차적인 수식과 충돌 가능성이 있는 세부 묘사를 줄이는 것을 뜻한다.

## 고성능 모델 승격 조건

- 같은 패널이 연속 2회 실패
- 필수 캐릭터 참조가 없거나 둘 이상으로 모호함
- 스토리보드 설명과 생성 프롬프트가 충돌
- QA 판정이 기준선 근처에서 반복적으로 흔들림
- 정책 또는 안전 문제 가능성
- 수정 범위가 여러 패널에 걸침

승격된 작업을 기다리는 동안에도 독립적인 `ready` 패널은 계속 처리한다.

## 안전한 첫 실행

실행기가 구현되면 EP002 전체를 바로 돌리지 않는다.

1. `--dry-run`으로 선택·복잡도·참조 자산만 확인한다.
2. `node scripts/run-panel-jobs.js --episode EPxxx --dry-run --max-jobs 3`으로
   일반 패널 최대 3장 또는 복잡 패널 1장의 다음 배치를 고른다.
3. JSON, 매니페스트, PNG 상태가 일치하는지 확인한다.
4. 실패 작업이 큐 전체를 막지 않는지 확인한다.
5. 검증 후에만 에피소드 전체 자율 실행으로 확대한다.
